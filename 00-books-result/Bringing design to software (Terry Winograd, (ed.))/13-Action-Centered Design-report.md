## 二、文献定位

该章提出 action-centered design，批判传统软件工程从需求规格到系统实现的线性模型会遮蔽用户真实工作与行动领域。它是把软件设计从产品中心和规格中心转向行动、实践、客户满意和持续沟通的重要文本。

## 三、核心内容摘要

Denning 与 Dargan 指出，软件工程自 1960 年代以来试图通过严格生命周期模型解决软件危机，但许多系统即使被交付也未被真正使用。问题并不只是规格错误，而是传统工程过程对用户实际生活和工作的“domain of action”存在盲区。作者主张软件设计应观察人们在某一领域中的重复行动、抱怨、期望和 breakdowns，再把这些行动过程连接到支持性软件技术中。

章节区分软件工程和人本设计。软件工程以产品、规格和交付为中心；人本设计则以满意客户、协作和持续沟通为中心。作者进一步提出软件架构应成为连接系统工程师和客户导向设计者的新学科，核心实践是通过行动模式映射把人的工作、承诺和软件功能连接起来。Quicken、MeetingMaker、Topic、Macintosh 等成功软件案例显示，优秀软件通常模仿用户熟悉行动，并支持用户曾经希望但无法手工完成的新行动。

## 五、核心概念

| 概念 | 英文 | 教材解释方向 |
|---|---|---|
| 行动中心设计 | action-centered design | 从用户行动领域出发，将重复行动和期望行动映射为软件支持。 |
| 行动领域 | domain of action | 人们在工作、娱乐、金融、教育等领域中持续参与的实践场域。 |
| 软件架构 | software architecture | 连接用户行动、软件功能和工程实现的设计学科。 |
| 行动模式映射 | pattern mapping | 把人类行动模式转化为软件功能和结构的方法。 |
| breakdowns | breakdowns | 用户行动过程中反复出现的中断、失败、抱怨和机会点。 |

## 六、教材使用建议

可用于“从需求规格到行动实践”的软件设计方法小节。它能帮助学生理解用户研究不只是访谈需求，而是观察人们真实行动、惯例、承诺、抱怨和使用后变化。

## 七、教学转化

课堂问题：为什么按照客户规格做出来的软件仍可能没人使用？作业可要求学生选择一个软件，画出其支持的用户行动流程，而不是功能列表。
